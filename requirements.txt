# was silently allowing pip to resolve a major version the project isn't
# tested on. It was also the largest single contributor to CI's
# "resolution-too-deep" (2026-07-14): <4.0.0 doubled the candidate set
# pip had to search (every 2.x release AND every 3.x release) on top of
# ccxt's and ctrader-open-api's own heavily pinned trees.

# Crypto exchange data (ccxt's own cryptography>=2.6.1 bound has no upper
# limit and will silently accept a vulnerable resolved version — floor it
# explicitly; audit finding, 2026-07-11). NO upper bound on cryptography:
# CI (PR #50, 2026-07-11) caught that the original <46.0.0 ceiling let pip
# resolve 45.0.7, which itself picked up 6 new CVEs (PYSEC-2026-35/36,
# CVE-2026-26007, GHSA-537c-gmf6-5ccf) patched only in 46.0.5+/48.0.1 —
# an artificial ceiling on a fast-patched security library actively blocks
# fixes. Floor-only, bumped to the version verified clean by pip-audit at
# fix time; re-bump the floor if pip-audit flags it again, don't re-add a
# ceiling.
#
# EXCEPT the top of the 4.5.x line right now: ccxt 4.5.65 hard-pins
# requests==2.34.2, which has no installable distribution for this
# environment (CI, 2026-07-14: "ResolutionImpossible... no matching
# distributions available for your environment: requests") — a broken
# release on ccxt's end, not a version we can satisfy by loosening our
# own requests range. Capped below it; bump back to <5.0.0 once a fixed
# ccxt patch release supersedes 4.5.65.
#
# Exact-pinned, not a range (CI, 2026-07-14): a range still left pip 65
# candidate ccxt releases (4.5.0-4.5.64) to multiply against every other
# loosely-bound package, which alone was enough to hit
# "resolution-too-deep" even after floor-constraining idna/urllib3/
# charset-normalizer/httpcore/h11. Exact-pinning the one package with by
# far the largest candidate set collapses that axis of the search
# entirely. Bump the exact version deliberately (not the range) when a
# fixed release supersedes 4.5.65.
ccxt==4.5.64
cryptography>=49.0.0

# cTrader (execution/ctrader_client.py) moved to requirements-ctrader.txt
# on 2026-07-14: ctrader-open-api's hard pin on pyOpenSSL==24.1.0 needs
# an older cryptography line than our security floor above allows, and
# resolving both in one pip invocation is what caused CI's
# "resolution-too-deep" (see requirements-ctrader.txt for the full
# explanation). Install it separately — required on the live VPS, safely
# omittable for CI/tests (the provider chain already falls back
# gracefully when it's absent).

# requests and httpx share these underneath (idna, urllib3/httpcore,
# charset-normalizer, h11) but neither is declared directly here, so pip
# had no floor to anchor on for any of them. Combined with ccxt's and
# ctrader-open-api's own heavily-pinned trees, that gave the resolver too
# large a combinatorial space and it hit "resolution-too-deep" (CI,
# 2026-07-14) before finding a solution — pip's own suggested fix is
# exactly this: add explicit lower bounds. Recent, permissive floors
# only, no ceiling beyond the next major.
idna>=3.7,<4.0
